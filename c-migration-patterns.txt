C++ TO C MIGRATION PATTERN DEFINITIONS

Purpose
=======

This file defines the standard C replacements to use while converting the
remaining C++ code to C.  New migration work should follow these patterns so
that the C codebase does not accumulate multiple styles for the same concept.

General API rules
=================

1. Public migration APIs should be plain C and should not require C++ syntax,
   C++ headers, C++ object lifetime rules, or C++ name mangling.
2. Prefer existing C modules in src/c/ and src/curses/ before adding a new
   helper type.
3. Ownership must be explicit.  Any function that allocates, borrows, takes, or
   releases storage should make that visible in its name, documentation, or
   surrounding type contract.
4. Initialization and cleanup must be explicit.  A type with owned resources
   should have init and destroy functions, or an equivalent create/free pair.
5. Error paths must be explicit.  Do not emulate exceptions with hidden global
   state or longjmp-style control flow unless a module already requires fatal
   termination.
6. Prefer concrete typed functions over clever generic abstractions.  Use macros
   only when they clearly reduce duplication without hiding ownership, cleanup,
   or evaluation order.

String values
=============

C++ pattern:
    std::string

C replacement:
    Use char * with an explicit length field when the string is mutable,
    slice-like, or may contain embedded NUL bytes.  Use existing NcmString
    utilities when they already cover the needed ownership and formatting
    behavior.

Rules:
    - Use const char * for borrowed NUL-terminated strings.
    - Use char * for owned NUL-terminated strings.
    - Use a length field, normally int32 len, when repeated strlen calls would
      be wasteful or when the string is not guaranteed to be NUL-terminated.
    - Do not return pointers to temporary buffers.
    - Formatting should use existing C string/format helpers or snprintf-style
      functions.

Dynamic arrays
==============

C++ pattern:
    std::vector<T>

C replacement:
    Use the existing dynamic array helpers, especially src/c/ncm_array.h, or a
    dedicated typed array struct when the element type needs custom ownership.

Rules:
    - Array structs should track pointer, count, and capacity.
    - Elements that own memory must be destroyed before the array storage is
      released.
    - Prefer append/reserve/clear/destroy functions over open-coded realloc at
      call sites.
    - Use int32 or int64 consistently with nearby code and expected sizes.

Associative maps
================

C++ pattern:
    std::map, std::unordered_map

C replacement:
    Use an explicit sorted array, a hash table, or a dedicated lookup struct.
    Choose the simplest representation that matches the data size and lookup
    frequency.

Rules:
    - For small fixed or rarely mutated sets, prefer sorted arrays and binary
      search.
    - For larger or frequently queried sets, use a hash table module or create a
      narrow dedicated lookup type.
    - Do not expose generic map internals through public APIs unless multiple
      modules truly share the same abstraction.

Optional values
===============

C++ pattern:
    std::optional<T>

C replacement:
    Use a bool has_x field plus a value field T x.

Rules:
    - The bool field should be checked before reading the value field.
    - Use names that make the relationship obvious, such as has_album and album.
    - If the optional value owns memory, cleanup should check the bool field or
      tolerate an empty value safely.

Errors and exceptions
=====================

C++ pattern:
    exceptions, throw, catch

C replacement:
    Use return codes, NcmError, or an explicit fatal call at the call site.

Rules:
    - Recoverable failures should return false, NULL, or a negative/enum error
      code, depending on the local API style.
    - Detailed recoverable diagnostics should use NcmError when available.
    - Fatal failures should be obvious at the call site and should not look like
      normal return paths.
    - Cleanup after failure should use explicit cleanup blocks or direct calls
      before returning.

Resource lifetime
=================

C++ pattern:
    RAII

C replacement:
    Use explicit init/destroy functions, create/free functions, and cleanup
    blocks.

Rules:
    - init should leave the object safe to destroy even after partial failure.
    - destroy should tolerate already-empty internal fields where practical.
    - A function with multiple owned resources should use a single cleanup path
      when that makes ownership clearer.
    - Avoid hidden allocation inside functions whose names do not imply it.

Object construction and destruction
===================================

C++ pattern:
    constructors, destructors

C replacement:
    Use explicit lifecycle functions.

Preferred names:
    type_init
    type_destroy
    type_create
    type_free
    type_reset
    type_clear

Rules:
    - init/destroy are preferred for stack or embedded objects.
    - create/free are preferred when the object itself is heap allocated.
    - reset/clear should define whether capacity is retained or released.

Inheritance and virtual dispatch
================================

C++ pattern:
    inheritance, virtual methods, override

C replacement:
    Use structs with function pointers, or explicit screen vtables for screen
    implementations.

Rules:
    - Shared base state should be stored in an embedded common struct.
    - Behavior should be represented by a small function table when dynamic
      dispatch is required.
    - Prefer direct function calls when the concrete type is known.
    - Screen implementations should follow the project screen abstraction rather
      than inventing per-screen dispatch styles.

Templates
=========

C++ pattern:
    templates

C replacement:
    Use typed macros only for simple repeated container operations, or write
    concrete typed functions.

Rules:
    - Prefer concrete typed functions when behavior is nontrivial.
    - Macros must not evaluate arguments more than once unless this is clearly
      documented and safe.
    - Public macro APIs should be avoided unless they are already part of the C
      foundation style.

Shared utility replacement modules
==================================

Use these C modules instead of the remaining shared C++ utility headers:

    utility/conversion.h          -> src/c/ncm_conversion.*
    utility/string_format.h       -> src/c/ncm_string_format.*
    utility/wide_string.h         -> src/c/ncm_wide_string.*
    utility/regex.h               -> src/c/ncm_regex.*
    utility/readline.h            -> src/c/ncm_readline.*
    utility/functional.h          -> src/c/ncm_functional.*
    utility/scoped_value.h        -> src/c/ncm_scoped_value.*
    utility/shared_resource.h     -> src/c/ncm_shared_resource.*
    utility/any_iterator.h        -> src/c/ncm_any_iterator.h
    utility/transform_iterator.h  -> src/c/ncm_transform_iterator.h

Do not add new wrappers for behavior already covered by cbase/ or src/c/.
Extend the existing C module instead.

Streams and formatting
======================

C++ pattern:
    iostream, fstream, stringstream

C replacement:
    Use FILE *, snprintf, and the existing string formatting helpers.

Rules:
    - File input should use FILE * or existing filesystem helpers.
    - Formatted output should use snprintf-style functions or existing buffer
      helpers.
    - Avoid fixed-size stack buffers for unbounded user/config data unless the
      limit is intentional and checked.
    - Do not introduce C++ stream-like wrappers in C.

Filesystem operations
=====================

C++ pattern:
    std::filesystem

C replacement:
    Use src/c/ncm_fs.* and POSIX filesystem calls.

Rules:
    - Prefer ncm_fs helpers for paths, existence checks, directory creation, and
      file operations when they already exist.
    - Use POSIX calls directly only when no project helper exists or when the
      helper would obscure required behavior.
    - Path ownership must be explicit: borrowed path, caller-owned path, or
      newly allocated path.

Regular expressions
===================

C++ pattern:
    std::regex

C replacement:
    Use src/c/ncm_regex.*.

Rules:
    - Compile regex objects explicitly and release them explicitly.
    - Keep matching/parsing code isolated from UI code where practical.
    - Prefer parser-specific tests for regex-heavy behavior.

Threads and async work
======================

C++ pattern:
    std::thread, std::future, std::async

C replacement:
    Use a pthread-based job abstraction, preferably src/c/ncm_job.*.

Rules:
    - Async jobs should have explicit ownership of input and output data.
    - Cancellation, completion, and cleanup should be part of the job contract.
    - Do not let worker threads access curses/UI state directly.
    - Results should be transferred back to the main/UI thread through explicit
      state or callback mechanisms.

Screen migration rules
======================

1. Convert each screen's C++ class state into a named C struct.
2. Convert methods into type-prefixed C functions.
3. Use the common C screen abstraction and vtable/function-table pattern for
   screen lifecycle, input handling, resize, refresh, and update behavior.
4. Keep curses/menu ownership explicit.  A screen that owns a window, menu, or
   buffer must destroy it in its cleanup path.
5. Keep model operations outside rendering functions when practical so they can
   be tested without curses.

Migration checklist for each converted C++ file
===============================================

1. Identify the C++ patterns used by the file.
2. Map each pattern to the replacement defined in this document.
3. Reuse existing src/c or src/curses modules where possible.
4. Add missing helper functions to the relevant C module instead of creating a
   one-off local abstraction.
5. Add or update tests when the converted behavior is nontrivial.
6. Remove the .cpp file from the build only after the C replacement is compiled
   and linked.
7. Run make all and make check after each migration batch.

Curses wrapper migration
========================

Use the native C curses types directly in converted code:

    NC::Key::Type     -> NcKey
    NC::Key::*        -> NC_KEY_*
    NC::Mouse::*      -> nc_mouse_*
    NC::Format        -> enum NcFormat
    NC::Format::*     -> NC_FORMAT_*
    NC::Scroll        -> enum NcScroll
    NC::Scroll::*     -> NC_SCROLL_*
    NC::Window        -> NcWindow
    NC::Buffer        -> NcBuffer
    NC::Color         -> NcColor
    NC::Border        -> NcBorder

Prefer nc_key_name(), nc_format_reverse(), nc_border_none(), and
nc_border_make() over reintroducing C++ helper namespaces.

Typed menu adapters
===================

Use src/curses/nc_app_menus.h for concrete replacements of current
NC::Menu<T> instantiations.

    NC::Menu<MPD::Song>              -> NcSongMenu
    NC::Menu<MPD::Item>              -> NcMpdItemMenu or NcBrowserEntryMenu
    NC::Menu<NcOutputsItem>          -> NcOutputMenu
    NC::Menu<NcmPlaylist>            -> NcPlaylistEntryMenu
    NC::Menu<MPD::MutableSong>       -> NcTagRowMenu
    NC::Menu<SEItem>                 -> NcSearchRowMenu
    NC::Menu<MediaLibrary::PrimaryTag> -> NcMediaLibraryTagMenu
    NC::Menu<MediaLibrary::AlbumEntry> -> NcMediaLibraryAlbumMenu
    NC::Menu<std::string>            -> NcEditorStringMenu
    NC::Menu<std::pair<string,string>> -> NcEditorPairMenu
    NC::Menu<RunnableItem<...>>      -> NcEditorActionMenu or NcEditorSortMenu
    NC::Menu<NC::Buffer>             -> NcEditorBufferMenu

Rules:
    - Store items by value in NcMenu through the typed adapter.
    - Attach draw, filter, selection, inactive, and activate callbacks through
      the underlying NcMenu returned by *_base().
    - Keep item ownership inside the adapter callbacks.
    - Do not add new NC::Menu<T> instantiations.

Configuration migration
=======================

Use src/settings.h as the C configuration contract. During the mixed-language
migration, existing C++ files include src/settings_legacy.h. New C code must not
include src/settings_legacy.h.

Rules:
    - Store owned strings as char * plus explicit int32 length and capacity.
    - Store parsed display formats as NcmFormatAst.
    - Store curses display fragments as NcBuffer.
    - Store formatted colors as NcFormattedColor and NcmFormattedColorArray.
    - Store optional screen values as a bool has_* flag plus the value.
    - Store durations as integer seconds.
    - Store vector-like configuration fields in typed C arrays.
    - Store song-column configuration in ColumnArray through
      column_init(), column_destroy(), column_array_init(),
      column_array_destroy(), and column_array_append().
    - Store screen switcher configuration in ScreenTypeArray through
      screen_type_array_init(), screen_type_array_destroy(), and
      screen_type_array_append().
    - Keep the temporary C++ settings bridge in settings_legacy.h/.cpp until
      the remaining C++ screens no longer use Configuration methods or
      std::vector-based settings.
    - Store lyrics fetcher definitions in NcmLyricsFetcherRegistry, not in
      C++ LyricsFetchers.

Configuration entry-point migration
===================================

Use src/configuration.h and src/configuration.c for the C configuration entry
point.

    expand_home(std::string &) -> expand_home(char **path, int32 *path_len,
                                              NcmError *error)
    configuration.cpp          -> configuration.c

Rules:
    - Expand only owned C strings allocated with ncm_malloc().
    - Pass config paths into configuration_read() as NcmStringViewArray.
    - Build default config and bindings paths with ncm_fs_user_config_dir()
      and ncm_fs_join().
    - Create directories with ncm_fs_mkdir_all().
    - Apply MPD host, port, and timeout through NcmMpdClient helpers.
    - Keep configuration_legacy.h/.cpp only while ncmpcpp.cpp and the old C++
      screens still require the C++ settings bridge.

Configuration parser follow-up
==============================

Use src/c/ncm_option_parser.h and src/c/ncm_option_parser.c for C config-file
option parsing.

    utility/option_parser.h line parsing -> ncm_option_parser_parse_line()
    utility yes_no()                     -> ncm_option_parser_yes_no()
    parser exceptions                    -> bool return plus NcmError

Rules:
    - Build default config and bindings paths with NcmBuffer, not std::string.
    - Discover XDG config paths with ncm_fs_user_config_dir().
    - Check existing migrated paths with ncm_fs_exists().
    - Create missing XDG config directories with ncm_fs_mkdir_all().
    - Keep detailed parser failures in NcmError instead of replacing them with
      generic failure messages.

Bindings migration
==================

Use src/bindings.h and src/bindings.c as the C binding configuration owner.

    Binding::ActionChain        -> NcmBinding.actions typed C array
    BindingsConfiguration       -> NcmBindingsConfiguration
    CommandsSet unordered_map   -> sorted NcmCommand array
    BindingsMap unordered_map   -> sorted NcmKeyBindings array
    Bindings.findCommand(name)  -> ncm_bindings_configuration_find_command()
    Bindings.get(key)           -> ncm_bindings_configuration_get()
    Bindings.generateDefaults() -> ncm_bindings_configuration_generate_defaults()
    Bindings.read(paths)        -> ncm_bindings_configuration_read_paths()

Rules:
    - Store runtime bindings as typed action descriptors, not C++ action
      objects.
    - Keep bindings_legacy.h/.cpp only while the C++ Actions runtime still owns
      action execution.
    - Do not add new std::unordered_map or std::vector storage to bindings.
